# Vue + Vuetify 4 + Capacitor アプリ実装のお作法

## このドキュメントの目的

ハイブリッドアプリをチームで実装する際の構造・規約を定める。
「動けばよい」ではなく「全画面が同じ作りで、誰が読んでも迷わない」状態を
目指すための指針。

---

## 1. ディレクトリ構成

```
src/
├── main.ts                 # エントリポイント(プラグイン登録のみ)
├── App.vue                 # ルートコンポーネント(レイアウト切替のみ)
├── layouts/                # レイアウト(画面の骨格)
│   ├── MainLayout.vue      #   通常画面用(アプリバー+ナビ)
│   └── BlankLayout.vue     #   ログイン画面等(骨格なし)
├── views/ (または pages/)   # ルーティング対象の画面コンポーネント
│   ├── HomeView.vue
│   └── SettingsView.vue
├── components/             # 再利用コンポーネント
│   ├── common/             #   アプリ全体共通(AppButton等)
│   └── home/               #   特定画面専用の部品
├── composables/            # ロジックの再利用(useXxx)
├── services/               # 外部接続の抽象化(API、Capacitorプラグイン)
│   ├── api/
│   └── native/             #   ネイティブ機能のラッパー(後述・重要)
├── stores/                 # Pinia(状態管理)
├── router/                 # vue-router定義
├── plugins/                # Vuetify等のプラグイン設定
│   └── vuetify.ts
└── assets/
```

原則:

- **views/ は「ルーティングで直接表示されるもの」だけ**。部品は components/ へ
- **1ファイルが300行を超えたら分割を検討**(部品化 or composable化)
- 特定画面でしか使わない部品は `components/<画面名>/` に置き、
  2画面以上で使うようになったら `common/` に昇格させる

## 2. レイアウトパターン(MainLayout)

### 基本構造

Vuetifyアプリは `v-app` を頂点に、レイアウト用コンポーネントで骨格を作る。
**骨格はレイアウトに集約し、各画面(views)はコンテンツだけを書く。**

```vue
<!-- src/layouts/MainLayout.vue -->
<template>
  <v-app>
    <v-app-bar :title="title" />

    <v-main>
      <router-view />
    </v-main>

    <!-- モバイルはボトムナビゲーションが定石 -->
    <v-bottom-navigation grow>
      <v-btn to="/" prepend-icon="mdi-home">ホーム</v-btn>
      <v-btn to="/search" prepend-icon="mdi-magnify">検索</v-btn>
      <v-btn to="/settings" prepend-icon="mdi-cog">設定</v-btn>
    </v-bottom-navigation>
  </v-app>
</template>
```

ポイント:

- **`v-main` の中に `router-view`** — `v-main` がアプリバーやナビの分の
  余白を自動計算してくれる。これを使わず自前でpaddingを書くのはNG
- スマホアプリは `v-navigation-drawer`(ハンバーガー)より
  `v-bottom-navigation`(下タブ)のほうがネイティブアプリらしいUXになる

### レイアウトの切替(route metaパターン)

ログイン画面など骨格が不要な画面のため、ルート定義のmetaで切り替える:

```typescript
// router/index.ts
const routes = [
  { path: '/login', component: LoginView, meta: { layout: 'blank' } },
  { path: '/',      component: HomeView },   // metaなし = main
]
```

```vue
<!-- App.vue -->
<template>
  <component :is="layoutComponent">
  </component>
</template>

<script setup lang="ts">
import { computed } from 'vue'
import { useRoute } from 'vue-router'
import MainLayout from '@/layouts/MainLayout.vue'
import BlankLayout from '@/layouts/BlankLayout.vue'

const route = useRoute()
const layoutComponent = computed(() =>
  route.meta.layout === 'blank' ? BlankLayout : MainLayout
)
</script>
```

### セーフエリア対応(Capacitor必須事項)

実機ではノッチ・ホームインジケータ領域にUIが食い込む。
CSSの `env(safe-area-inset-*)` で対応する:

```css
/* レイアウトまたはグローバルCSS */
.v-bottom-navigation {
  padding-bottom: env(safe-area-inset-bottom);
}
```

ブラウザでは値が0になるだけなので、開発初期から入れておいて害はない。
**後付けすると全画面の調整になるため、レイアウト側に最初から仕込む。**

## 3. 共通部品(コンポーネント)の作り方

### ラッパーコンポーネント方針

Vuetifyコンポーネントを直接使い散らかすと、デザイン変更時に全画面修正に
なる。**アプリ標準の使い方を固定したラッパーを common/ に用意する**:

```vue
<!-- components/common/AppDialog.vue -->
<template>
  <v-dialog v-model="model" :max-width="width" :persistent="persistent">
    <v-card>
      <v-card-title>{{ title }}</v-card-title>
      <v-card-text><slot /></v-card-text>
      <v-card-actions>
        <v-spacer />
        <slot name="actions" />
      </v-card-actions>
    </v-card>
  </v-dialog>
</template>

<script setup lang="ts">
const model = defineModel<boolean>({ required: true })
withDefaults(defineProps<{
  title: string
  width?: number
  persistent?: boolean
}>(), { width: 400, persistent: false })
</script>
```

規約:

- 命名は `App` プレフィックス(`AppButton`, `AppDialog`, `AppConfirm`)。
  Vuetify(`v-`)・画面部品と一目で区別できる
- **props / emits は必ずTypeScriptで型定義**(`defineProps<{...}>()`)
- 拡張性は slot で確保し、props を増やしすぎない
- ただし**何でもラップしない**。「アプリ独自の標準形がある」ものだけ
  ラップする(単純な `v-btn` まで包むのは過剰)

### Vuetifyのdefaultsで「ラップ未満」の統一をする

「色とサイズだけ統一したい」レベルなら、ラッパーを作らず
**Vuetifyのグローバルdefaults**で済ませる:

```typescript
// plugins/vuetify.ts
export default createVuetify({
  defaults: {
    VBtn: { color: 'primary', variant: 'flat' },
    VTextField: { variant: 'outlined', density: 'comfortable' },
  },
  theme: {
    defaultTheme: 'light',
    themes: {
      light: {
        colors: { primary: '#1976D2', secondary: '#424242' },
      },
    },
  },
})
```

> 使い分け: **見た目の既定値 → defaults / 構造・振る舞いの標準形 → ラッパー**

テーマカラーは必ず `plugins/vuetify.ts` に集約し、
コンポーネント内に色コードを直書きしない(`color="primary"` を使う)。

## 4. ロジックの置き場所

### composables(useXxx)

複数コンポーネントで使うリアクティブなロジックは composable に切り出す:

```typescript
// composables/useConfirm.ts — 確認ダイアログの共通ロジック例
export function useConfirm() {
  const isOpen = ref(false)
  const message = ref('')
  let resolver: ((v: boolean) => void) | null = null

  function confirm(msg: string): Promise<boolean> {
    message.value = msg
    isOpen.value = true
    return new Promise(res => { resolver = res })
  }
  function answer(v: boolean) { isOpen.value = false; resolver?.(v) }

  return { isOpen, message, confirm, answer }
}
```

規約: ファイル名・関数名とも `useXxx`。viewsに直接ロジックを書き始めたら
composable化を検討する。

### stores(Pinia)

画面をまたいで保持する状態(ログインユーザー、設定値等)はPiniaへ。
**「画面内で完結する状態」までstoreに入れない**(ref/composableで足りる)。

## 5. Capacitor(ネイティブ機能)のお作法

### 最重要: プラグインを直接呼ばず services/native/ でラップする

Capacitorプラグインを画面から直接呼ぶと、
ブラウザ開発時(=日常の9割)に動かない・テストできない問題が起きる。
**必ずサービス層を挟み、Web実行時のフォールバックを内包させる:**

```typescript
// services/native/camera.ts
import { Capacitor } from '@capacitor/core'
import { Camera, CameraResultType } from '@capacitor/camera'

export async function takePhoto(): Promise<string | null> {
  if (!Capacitor.isNativePlatform()) {
    // Web実行時のフォールバック(input[type=file]を使う、ダミーを返す等)
    return await pickPhotoViaFileInput()
  }
  const photo = await Camera.getPhoto({
    resultType: CameraResultType.Uri,
    quality: 80,
  })
  return photo.webPath ?? null
}
```

この方針の効果:

- ブラウザのデバイスモードで開発を続けられる(手順書③のワークフロー前提)
- テスト時にservices層だけモックすればよい(テスト編⑥と連動)
- プラグイン差し替え・Capacitorメジャーアップ時の影響範囲がservicesに閉じる

### Androidの「戻る」ボタン対応(必須)

何もしないと戻るボタンでアプリが即終了する。アプリ起動時に1箇所で制御する:

```typescript
// services/native/backButton.ts
import { App } from '@capacitor/app'
import type { Router } from 'vue-router'

export function setupBackButton(router: Router) {
  App.addListener('backButton', ({ canGoBack }) => {
    if (canGoBack) router.back()
    else App.exitApp()   // ルート画面では終了(または確認ダイアログ)
  })
}
```

### その他の初期設定リスト

| 項目 | プラグイン | やること |
|---|---|---|
| スプラッシュ | @capacitor/splash-screen | 表示時間・背景色を設定し、起動完了で `hide()` |
| ステータスバー | @capacitor/status-bar | テーマと色を合わせる |
| キーボード | @capacitor/keyboard | 入力欄が隠れる問題への resize 設定 |
| アプリ状態 | @capacitor/app | バックグラウンド復帰時の再読込制御 |

### 環境変数

APIエンドポイント等は `.env` ファイルで管理(Viteの規約で `VITE_` プレフィックス
必須):

```
.env.development   → VITE_API_BASE_URL=http://localhost:8080
.env.production    → VITE_API_BASE_URL=https://api.example.com
```

`capacitor.config.ts` のライブリロード用 `server.url` は
**コミットしない**(手順書③参照)。環境で切り替えるなら
`capacitor.config.ts` をTSの条件分岐で書く。

## 6. 画面実装の標準形(テンプレート)

新規画面はこの形から始める:

```vue
<!-- views/SampleView.vue -->
<template>
  <v-container>
    <!-- ローディング・エラー・正常系の3状態を必ず扱う -->
    <v-skeleton-loader v-if="loading" type="list-item@3" />
    <v-alert v-else-if="error" type="error" :text="error" />
    <template v-else>
      <!-- コンテンツ -->
    </template>
  </v-container>
</template>

<script setup lang="ts">
import { ref, onMounted } from 'vue'

const loading = ref(true)
const error = ref<string | null>(null)

onMounted(async () => {
  try {
    // データ取得
  } catch (e) {
    error.value = '読み込みに失敗しました'
  } finally {
    loading.value = false
  }
})
</script>
```

規約:

- `<script setup lang="ts">` + Composition API で統一(Options API禁止)
- **3状態(loading / error / 正常)を省略しない** — モバイルは回線が不安定
- ルートは可能な限り遅延読込: `component: () => import('@/views/...')`

## 7. 規約まとめ(チェックリスト)

- [ ] 画面の骨格はレイアウトに、コンテンツはviewsに分離されているか
- [ ] `v-main` + `router-view` の構造を崩していないか
- [ ] セーフエリア対応がレイアウトに入っているか
- [ ] 色コードの直書きがないか(テーマ経由か)
- [ ] Capacitorプラグインを画面から直接呼んでいないか(services経由か)
- [ ] Web実行時のフォールバックがあるか
- [ ] 戻るボタン制御が入っているか
- [ ] props/emitsに型が付いているか
- [ ] loading/error/正常の3状態を扱っているか
- [ ] ライブリロード設定をコミットしていないか
