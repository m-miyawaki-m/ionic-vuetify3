# Oracle Linux 9 コンテナイメージ変更手順(JDK 21 + Node.js 22)

## 目的

Oracle Linux 9 ベースのコンテナイメージを以下のように構成変更する。

| 項目 | 現在 | 変更後 |
|---|---|---|
| JDK | 17 | 21 |
| Gradle | 9 | 9(JDK 21対応のため変更不要 / 不要なら削除) |
| Node.js | なし | 22(npm はNode.jsに同梱。Node.js 22 同梱のnpmは10系) |

> **補足**: 「npm 22」は「Node.js 22」と解釈。npm自体のバージョンは現在10系。

## 基本方針

既存イメージに手作業で追加するのではなく、**Dockerfileを修正してイメージを再ビルドする**。

---

## 事前確認: 現在のイメージの把握

- **Dockerfileがある場合**: Dockerfileを修正して再ビルド(通常はこちら)
- **イメージしかない場合**: `docker history <イメージ名>` でレイヤー確認。
  そのイメージを `FROM` にして上乗せするか、Dockerfileを再作成する

現在のJDK 17の導入方式を確認:

```bash
docker run --rm <イメージ名> sh -c "java -version; which java; rpm -qa | grep -i -e jdk -e java"
```

→ dnf(rpm)導入か、tarball配置かで差し替え方法が変わる。

---

## 手順

### Step 1: JDK 21 への差し替え

#### パターンA: dnf でインストールされている場合(推奨)

Oracle Linux 9 標準リポジトリの `java-21-openjdk` を利用。

```dockerfile
# 変更前
RUN dnf install -y java-17-openjdk-devel && dnf clean all

# 変更後
RUN dnf install -y java-21-openjdk-devel && dnf clean all
```

ポイント:

- ビルド用途なら `-devel`(javac入り)が必要。実行専用なら
  `java-21-openjdk-headless` で十分(イメージも小さい)
- `JAVA_HOME` のパス更新を忘れない:

  ```dockerfile
  ENV JAVA_HOME=/usr/lib/jvm/java-21-openjdk
  ENV PATH=$JAVA_HOME/bin:$PATH
  ```

- 旧JDKの明示削除は `dnf remove -y java-17-openjdk\*` でも可能だが、
  最初から21だけ入れる形に書き換えるほうがクリーン

#### パターンB: Oracle JDK / tarball 配置の場合

- ダウンロードURL・バージョン番号・展開先パスを差し替え
- Oracle Linux なら Oracle JDK は `dnf install jdk-21`(Oracleのyumリポジトリ)でも可
- OpenJDKで問題なければパターンAが管理は楽

### Step 2: Node.js 22 の追加

#### 方法a: dnf モジュール(標準リポジトリ、推奨)

```dockerfile
RUN dnf module enable -y nodejs:22 && \
    dnf install -y nodejs && \
    dnf clean all
```

AppStream リポジトリに含まれるため追加リポジトリ不要。

#### 方法b: NodeSource リポジトリ(より新しいマイナーバージョンが必要な場合)

```dockerfile
RUN curl -fsSL https://rpm.nodesource.com/setup_22.x | bash - && \
    dnf install -y nodejs && \
    dnf clean all
```

- npm は Node.js に同梱されるため個別インストール不要
- 確認: `RUN node --version && npm --version`

### Step 3: Gradle 9 の扱い

- **残す場合**: Gradle 9 は JDK 21 を完全サポート → Gradle自体の変更不要
- **外す場合**: Dockerfile内のGradleインストール部分を削除
- プロジェクト側の対応(別途必要): `build.gradle` の toolchain 設定を21へ

  ```groovy
  java {
      toolchain {
          languageVersion = JavaLanguageVersion.of(21)
      }
  }
  ```

### Step 4: 再ビルドと動作確認

```bash
docker build -t myapp:jdk21-node22 .

# 確認
docker run --rm myapp:jdk21-node22 java -version    # openjdk 21.x
docker run --rm myapp:jdk21-node22 node --version   # v22.x
docker run --rm myapp:jdk21-node22 gradle --version # 残した場合
```

- `latest` を上書きせず新タグで並行稼働確認してから切り替えるのが安全

## プロキシ環境でのビルド

### 前提

WSL2側の3層プロキシ設定が済んでいれば、ビルド時の通信はすべてプロキシ経由でつながる。

### ビルド中の通信と設定の対応関係

| ビルド中の通信 | 実行場所 | カバーする設定 |
|---|---|---|
| `FROM oraclelinux:9` のpull | Dockerデーモン | `/etc/docker/daemon.json` の `proxies` |
| `RUN dnf install ...` | コンテナ内 | `~/.docker/config.json` の `proxies.default` |
| `RUN curl ...` | コンテナ内 | 同上 |
| `npm install` 等 | コンテナ内 | 同上 |

- 今回特に重要なのは **`~/.docker/config.json`(コンテナ/ビルド用)**
- 設定済みなら `docker build` 時にプロキシ環境変数が自動注入される
- Dockerfile に `ENV http_proxy=...` を書く必要はない
  (書かないほうがイメージにプロキシ情報が焼き込まれず安全)

### 事前の動作確認コマンド

再ビルド前に単体で確認できる:

```bash
# 環境変数が注入されているか
docker run --rm oraclelinux:9 env | grep -i proxy

# コンテナ内から外部に出られるか
docker run --rm oraclelinux:9 curl -sI https://yum.oracle.com | head -1

# dnf が通るか
docker run --rm oraclelinux:9 dnf -q repolist
```

→ 3つとも通れば再ビルドもほぼ確実に通る。

### つながらない場合の例外パターン

#### 1. dnf がプロキシを拾わない

dnf は環境変数 `https_proxy`(小文字)を参照するが、挙動に揺れがある事例あり。
dnf だけタイムアウトする場合は明示設定が確実:

```dockerfile
RUN echo "proxy=http://proxy.example.com:8080" >> /etc/dnf/dnf.conf
```

※ イメージにプロキシ情報が残るため、社外配布イメージでは最終ステージで削除するか
`--mount=type=secret` 等を検討

#### 2. SSLインスペクション(HTTPS復号)型プロキシ

プロキシには届くのに証明書エラーになる
(`curl: (60) SSL certificate problem` 等)。
→ 社内CA証明書をコンテナに追加:

```dockerfile
COPY company-ca.crt /etc/pki/ca-trust/source/anchors/
RUN update-ca-trust
```

npm は独自に証明書検証するため、必要に応じて `npm config set cafile ...` も設定。

#### 3. NO_PROXY の過不足

- 社内ミラー(Nexus/Artifactory等)宛がプロキシ経由になって失敗 → NO_PROXYに追加
- 外部宛先が誤ってNO_PROXYに入っていて直接接続を試みて失敗 → NO_PROXYから除外
- PACファイルで `DIRECT` を返す条件と NO_PROXY の内容が一致しているか確認

### 障害切り分けフロー

1. **ベースイメージのpullで失敗** → デーモン設定(`daemon.json`)
   → `docker info | grep -i proxy` で確認
2. **`RUN` 命令内の通信で失敗** → `config.json`
   → `docker run --rm ... env | grep -i proxy` で確認
3. **プロキシには届くが証明書エラー** → SSLインスペクション → CA証明書追加
4. **特定の宛先だけ失敗** → NO_PROXY の過不足

## その他の注意点

- **JDK 17→21 のアプリ互換性**: コンテナ構成変更とは別に、アプリケーションコードが
  JDK 21 で動くかの確認が必要(非推奨API、ライブラリ対応状況など)。
  特に古いフレームワーク(Spring Boot 2系など)は JDK 21 非対応の場合あり
- **マルチステージビルド**: ビルド用(JDK+Gradle+Node)と実行用(JREのみ)を分けると
  本番イメージを大幅に小型化できる。今回の変更を機に検討の価値あり
- **`dnf clean all`** を各 RUN の最後に入れてキャッシュを残さない
