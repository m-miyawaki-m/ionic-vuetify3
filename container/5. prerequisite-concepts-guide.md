# 前提知識編: 環境構築手順を理解するための概念ガイド

## このドキュメントの目的

環境構築の4つの手順書(WSL2+Docker構築 / コンテナイメージ更新 /
アプリ開発環境 / VS Code設定管理)は「何をするか」を示すが、
トラブル時に自力で対処するには「なぜそうなるのか」の理解が必要になる。

このドキュメントでは、手順の背後にある概念を解説する。

| 章 | 概念 | 主に関係する手順書 |
|---|---|---|
| 1 | 仮想化の階層 | 全部(特にエミュレータが動かせない理由) |
| 2 | コンテナの基本概念 | ②イメージ更新、①プロキシ3層設定 |
| 3 | ネットワークとプロキシ | ①プロキシ設定、③ライブリロード |
| 4 | Linuxの基礎概念 | ①systemd設定、②dnf操作 |
| 5 | 開発ツールチェーン | ②JDK/Node、③Capacitor |
| 6 | VS Codeリモート開発の構造 | ④拡張管理 |

## 1. 仮想化の階層 — すべての土台

### 今回の構成は「入れ子の仮想化」でできている

```
物理PC(Windows)
 └─ Hyper-V(Windowsのハイパーバイザ)
     └─ WSL2(軽量Linux仮想マシン)
         └─ Docker(コンテナ = プロセスの隔離。仮想マシンではない)
             └─ ※ここでさらに仮想マシン(エミュレータ)を動かすのは無理がある
```

### WSL1 と WSL2 の違い

- **WSL1**: WindowsカーネルがLinuxのシステムコールを変換する方式。
  本物のLinuxカーネルは存在しない
- **WSL2**: **Hyper-V上で本物のLinuxカーネルを動かす軽量仮想マシン**。
  Dockerが動くのはこちらだけ(DockerはLinuxカーネルの機能を使うため)

→ 手順書①で「VERSION が 2 であること」を確認するのはこのため。

### 仮想マシンとコンテナは別物

| | 仮想マシン(VM) | コンテナ |
|---|---|---|
| 持っているもの | OS丸ごと(カーネル含む) | プロセス+ファイルシステムのみ |
| カーネル | 自前 | **ホスト(WSL2)と共有** |
| 起動 | 数十秒〜 | 数秒以下 |
| 例 | WSL2、Androidエミュレータ | Dockerコンテナ |

**コンテナは「軽い仮想マシン」ではなく「隔離されたプロセス」**。
この違いが分かると、次のことが自然に理解できる:

- コンテナにはOSの「起動」がないため、systemdのようなサービス管理が
  そのままでは動かない
- コンテナの中身(Oracle Linux 9)とWSLのディストリ(Ubuntu)が違っても
  問題ない — どちらも同じWSL2カーネルの上で動くプロセスにすぎない

### なぜエミュレータはコンテナで動かせないのか

Androidエミュレータは**仮想マシン**であり、CPUの仮想化支援(KVM)を必要とする。
コンテナの中でこれをやろうとすると:

```
Hyper-V → WSL2 → (KVM) → エミュレータ   ← 仮想化の三重入れ子
```

ネスト仮想化を有効にすれば技術的には可能だが、性能劣化・GPU問題・
サポート外という理由で実用に耐えない。
→ 手順書③で「エミュレータはWindows側」と結論づけた根拠。

## 2. コンテナの基本概念

### イメージとコンテナの関係

```
Dockerfile ──(docker build)──→ イメージ ──(docker run)──→ コンテナ
 (設計図)                      (読み取り専用の型)          (実行中の実体)
```

- **イメージ**: ファイルシステムのスナップショット。変更できない。配布できる
- **コンテナ**: イメージから起動した実行中(または停止中)の実体。
  使い捨てが基本

ここから導かれる重要な原則:

> **コンテナ内で手作業した変更は、コンテナを消すと失われる。
> 永続させたい変更はDockerfileに書いてイメージを作り直す。**

→ 手順書②で「既存イメージに手で入れるのではなくDockerfileを修正して
再ビルド」とした根拠。

### レイヤ構造

イメージはDockerfileの命令(RUN等)ごとに積み重なる「レイヤ」でできている。

- `docker history` でレイヤを確認できる(手順書②の事前確認)
- 各 `RUN` の最後に `dnf clean all` を入れるのは、
  **そのレイヤにキャッシュが焼き込まれて消せなくなる**のを防ぐため

### デーモンとクライアントの分離 —「プロキシ3層」の正体

`docker` コマンドは命令を出すだけのクライアントで、
実際の仕事は **dockerd(デーモン)** という別プロセスが行う。

```
あなたのシェル(docker CLI)──→ dockerd(デーモン)──→ コンテナ(別の隔離空間)
```

**通信を行う「プロセス」が3つの別々の場所で動いている**ため、
プロキシ設定も3箇所必要になる:

| 通信の主体 | 例 | 設定場所 |
|---|---|---|
| シェルのプロセス | curl、git | 環境変数(~/.bashrc) |
| dockerd | docker pull | /etc/docker/daemon.json |
| コンテナ内のプロセス | RUN dnf install | ~/.docker/config.json |

> **環境変数は「プロセスごと」に持つもので、システム全体の設定ではない。**
> シェルに設定した環境変数は、別プロセスであるdockerdにも、
> 隔離されたコンテナ内にも届かない — これが3層必要な本質的理由。

## 3. ネットワークとプロキシ

### フォワードプロキシとは

社内ネットワークでは、外部への通信を**プロキシサーバー経由に強制**する
構成が一般的(セキュリティ監視・フィルタリングのため)。

```
社内PC ──→ プロキシサーバー ──→ インターネット
        (ここを通らないと外に出られない)
```

各ツール(ブラウザ、curl、dnf、npm…)は「プロキシを使え」と
**個別に教えてもらわないと知らない**。ブラウザはOS設定を見るが、
Linuxのコマンドラインツールは見ない — これが設定作業が多い理由。

### PAC(プロキシ自動構成)スクリプト

「宛先に応じてプロキシを使い分ける」ロジックをJavaScriptで書いたもの。

- **解釈できるのは実質ブラウザだけ**。curl・dnf・Dockerは解釈できない
- そのためPACの中身から固定のプロキシURLを取り出して設定する
  (手順書①の方針)
- PACで `DIRECT`(プロキシを使わない)と書かれた宛先 = `NO_PROXY` に
  設定すべき宛先

### NO_PROXY の意味

「この宛先にはプロキシを**使わずに**直接つなぐ」という除外リスト。

- localhost をプロキシに送ると、プロキシサーバーの localhost(=別マシン)を
  探しに行ってしまい失敗する → localhost は必ず NO_PROXY に入れる
- 社内サーバー(社内Nexus等)も通常はプロキシ不要 → NO_PROXY に入れる

### 「localhost」は場所によって意味が変わる

localhost(127.0.0.1)は「**今このプロセスが動いているマシン自身**」を指す。
仮想化の階層をまたぐと指す相手が変わる:

| どこから見た localhost | 指す相手 |
|---|---|
| Windowsのアプリから | Windows自身 |
| WSL2の中から(NATモード) | WSL2自身(Windowsではない!) |
| コンテナの中から | コンテナ自身 |
| Androidエミュレータの中から | エミュレータ自身 |

このズレを埋める仕組み:

- **WSL2のlocalhost転送**: Windowsから `localhost:5173` でWSL内の
  サーバーに届く(Windows→WSL方向の特別な転送)
- **mirroredモード**: WSLとWindowsでlocalhostを共有する新しい方式
- **10.0.2.2**: Androidエミュレータから見た「ホストマシン(Windows)」の
  特別なアドレス(手順書③のライブリロード設定の正体)

### SSLインスペクションと証明書

HTTPS通信は暗号化されているため、プロキシは中身を見られない。
そこで一部の社内プロキシは**通信をいったん復号して再暗号化**する
(SSLインスペクション)。

- このとき通信相手の証明書が「社内CA(認証局)が発行したもの」に
  すり替わる
- 社内PCのブラウザは社内CA証明書がインストール済みなので気づかないが、
  **コンテナの中には社内CA証明書がない**ため「証明書が信頼できない」
  エラーになる
- 対処 = 社内CA証明書をコンテナに追加して「信頼してよい」と教える
  (手順書②の `update-ca-trust`)

## 4. Linux の基礎概念

### systemd — サービス管理の仕組み

Linuxで常駐プロセス(デーモン)の起動・停止・自動起動を管理する仕組み。
`systemctl start docker` の `systemctl` がその操作コマンド。

- 通常のLinuxでは標準装備だが、**WSL2では歴史的にオフがデフォルト**
  だった(現在は設定で有効化できる)
- Dockerデーモン(dockerd)は常駐プロセスなので、systemdがあると
  自動起動などの管理が楽になる
- → 手順書①で `/etc/wsl.conf` に `systemd=true` を書いた理由

### パッケージ管理: dnf と apt

Linuxでは、ソフトウェアを「リポジトリ」(パッケージの配布サーバー)から
パッケージ管理コマンドで導入する。

| 系統 | コマンド | 使うディストリ |
|---|---|---|
| Red Hat系 | **dnf** | Oracle Linux、RHEL、Fedora |
| Debian系 | **apt** | Ubuntu、Debian |

今回の構成では**両方が登場する**ことに注意:

- WSL2の Ubuntu → apt(手順書①のDockerインストール)
- コンテナの Oracle Linux 9 → dnf(手順書②のJDK/Nodeインストール)

関連概念:

- **リポジトリの追加**: 標準リポジトリにないもの(Docker本体、NodeSource版
  Node.js等)は、配布元のリポジトリを登録してから入れる
- **GPG鍵**: パッケージが改ざんされていないことを検証する署名鍵。
  リポジトリ追加時に一緒に登録する(手順書①の `docker.asc` の正体)
- **モジュールストリーム(dnf module)**: 同じパッケージの複数バージョン系統
  (Node.js 18/20/22等)を切り替える仕組み(手順書②の `nodejs:22`)

### 環境変数

プロセスに渡される「名前=値」のペア。重要な性質:

1. **プロセスごとに独立** — 親から子へコピーされるが、横のプロセスには
   伝わらない(プロキシ3層問題の根本)
2. **設定ファイルはただの起動時スクリプト** — `~/.bashrc` に書いた export は
   「新しいシェルを開いたとき」に実行されるだけ。書いた瞬間に
   全プロセスへ反映されるわけではない
3. 慣習として大文字(`HTTP_PROXY`)と小文字(`http_proxy`)の両方を見る
   ツール・片方しか見ないツールが混在する → 両方設定するのが定石

## 5. 開発ツールチェーン

### JDK / JRE / Gradle / Node / npm の関係

```
Javaの世界                      JavaScriptの世界
 JDK(開発キット: javac含む)      Node.js(実行環境)
  ⊃ JRE(実行環境のみ)            └ npm(同梱のパッケージ管理ツール)
 Gradle(ビルドツール。JDKの上で動く)
```

- **JDKとJREの違い**: コンパイル(javac)するならJDK、動かすだけならJRE。
  手順書②で `-devel`(JDK相当)と `-headless`(JRE相当)を
  使い分けたのはこのため
- **GradleはJavaアプリ**: Gradle自体がJDKの上で動くため、
  「Gradleのバージョン」と「対応するJDKバージョン」に組み合わせ制約がある
- **npmはNode.jsに同梱**: npm単体でバージョンを選んで入れるものではない。
  「npm 22」という表記が実際は「Node.js 22」を指していたのはこのため
- バージョン番号は製品ごとに独立(JDK 21、Node 22、npm 10 が同時に存在する)

### ハイブリッドアプリと Capacitor

モバイルアプリの作り方は大きく3系統:

| 方式 | 説明 | 例 |
|---|---|---|
| ネイティブ | OS専用言語で開発 | Kotlin/Swift |
| クロスプラットフォーム | 独自UIをネイティブ描画 | Flutter、React Native |
| **ハイブリッド** | **Webアプリをアプリの皮で包む** | **Capacitor**、Cordova |

Capacitorの仕組み:

```
Androidアプリ(ネイティブの外殻)
 └─ WebView(アプリ内ブラウザ)
     └─ Vue + Vuetify のWebアプリ(HTML/CSS/JS)が表示される
 └─ ネイティブブリッジ(カメラ・通知等のOS機能をJSから呼べるようにする)
```

ここから導かれる理解:

- **中身はWebアプリ**だから、日常の開発・確認はブラウザでできる
  (手順書③のワークフローの根拠)
- ブラウザで確認**できない**のはネイティブブリッジ部分(カメラ等)と
  WebViewの固有挙動 → そこだけエミュレータで確認
- **ライブリロード**は「WebViewの表示先を開発サーバーのURLに
  差し替える」だけの仕組み(`server.url` 設定の正体)
- Androidの外殻をビルドするのに JDK + Gradle + Android SDK が必要
  (WebアプリなのにJavaツールチェーンが要る理由)

### ビルドとは何か / dev サーバーとは何か

- **ビルド(`npm run build`)**: ソースコードを配布可能な静的ファイル
  (HTML/CSS/JS)に変換すること。成果物は `dist/` に出る
- **開発サーバー(`npm run dev`)**: ビルドせずにソースをその場で変換・配信し、
  ファイル変更を即座にブラウザへ反映(ホットリロード)する仕組み
- Capacitorの `cap sync` は「ビルド成果物をAndroidプロジェクトに
  コピーする」作業

## 6. VS Code リモート開発の構造

### クライアント/サーバー分離

VS Codeのリモート開発は「画面はWindows、処理は向こう側」という分離で
成り立つ。

```
[Windows]                          [WSL2/コンテナ/SSH先]
 VS Code(クライアント)   ←──→     VS Code Server(自動設置される)
  画面・入力・テーマ                 言語解析・Lint・デバッグ・ターミナル
```

この構造から自然に導かれること:

- **拡張機能が2種類に分かれる**: 画面に関わる拡張(テーマ等)はクライアント側、
  コードに関わる拡張(Volar、ESLint等)はサーバー側に入る
  (手順書④のマトリクスの根拠)
- **言語系拡張をWindowsに入れても意味がない**: 解析対象のファイルも
  Node.jsもコンテナ側にあるため
- VS Code Server は接続時に**自動でダウンロード・設置**される
  → プロキシ環境ではこのダウンロードが失敗ポイントになる(手順書④)

### Dev Containers が解決する問題

「環境構築手順書どおりにやったのに人によって動かない」問題の根本は、
**環境がドキュメント(自然言語)で管理されている**こと。

Dev Containers は環境定義をコード(devcontainer.json + Dockerfile)にして
Gitで管理する。これは **Infrastructure as Code** の考え方の開発環境版:

- 手順書は陳腐化するが、コードは実行されるため乖離しない
- `git clone` → 「Reopen in Container」で誰でも同じ環境
- 環境の変更がプルリクエストでレビューできる

## まとめ: 概念マップ

```
仮想化の階層         → なぜWSL2が必要か / なぜエミュレータは外か
  └ VMとコンテナの違い → なぜDockerfileで管理するか / systemdの特殊事情

プロセスと環境変数    → なぜプロキシ設定が3層必要か
  └ localhostの相対性 → なぜ 10.0.2.2 や localhost転送が出てくるか
  └ PAC・SSL復号      → なぜ固定URL化・CA証明書追加が必要か

ツールチェーンの関係  → なぜJDKとGradleに組み合わせ制約があるか
  └ ハイブリッドアプリ → なぜブラウザで開発しエミュレータは最終確認か

クライアント/サーバー → なぜ拡張を入れる場所が分かれるか
  └ 環境のコード化    → なぜdevcontainer.jsonをGit管理するか
```

手順の1つ1つは、これらの概念の帰結にすぎない。
トラブル時は「どの階層・どのプロセスで起きているか」から考えると、
4つの手順書のどこを見ればよいかが判断できる。
