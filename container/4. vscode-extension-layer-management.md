# VS Code 設定レイヤと拡張機能の導入・管理方針

## 前提: VS Code のリモートアーキテクチャ

Dev Containers / WSL 接続時、VS Code は2つに分かれて動作する。

```
[Windows]                      [WSL2 / コンテナ]
 VS Code本体(UI)    ←──→     VS Code Server(自動インストール)
  - 画面描画・入力              - 言語解析・Lint・デバッガ実行
  - テーマ、キー操作            - ファイルアクセス、ターミナル
  - UI拡張が動く                - ワークスペース拡張が動く
```

拡張機能は種類に応じて「UI側」「リモート側」のどちらで動くかが決まっており、
VS Code がある程度自動判別する(リモート接続中は「Install in Container」の
ように適切な側を提示)。

## 基本方針

| レイヤ | 方針 |
|---|---|
| Windows(ローカル) | 見た目・操作系・リモート接続用の拡張のみ |
| WSL2(Ubuntu) | **原則何も入れない**(コンテナの土台に徹する) |
| コンテナ | 開発に必要な拡張をすべて入れ、**devcontainer.json でコード管理** |

## 設定レイヤ・マトリクス

凡例: ◎=ここで管理(主担当) / ○=必要に応じて / −=入れない・対象外

| 設定項目 | Windows(ローカル) | WSL2(Ubuntu) | コンテナ |
|---|:---:|:---:|:---:|
| **■ 拡張機能** | | | |
| Dev Containers / WSL拡張(リモート接続) | ◎ | − | − |
| テーマ・アイコンテーマ | ◎ | − | − |
| キーマップ(Vim等) | ◎ | − | − |
| Vue - Official(Volar) | − | − | ◎ |
| ESLint / Prettier | − | − | ◎ |
| EditorConfig | − | − | ◎ |
| Java / Gradle系(android配下編集用) | − | − | ○ |
| GitLens | − | − | ○ |
| Docker拡張(Dockerfile編集) | − | − | ○ |
| ShellCheck等(WSL直接作業する場合のみ) | − | ○ | − |
| **■ 設定(settings)** | | | |
| フォントサイズ・テーマ等の見た目 | ◎(User設定) | − | − |
| フォーマッタ・Lint・インデント幅 | − | − | ◎(.vscode/settings.json) |
| コンテナ内の既定値(シェル指定等) | − | − | ◎(devcontainer.json) |
| **■ 管理方法** | | | |
| Settings Sync(アカウント同期) | ◎ | − | − |
| devcontainer.json(Git管理・必須拡張) | − | − | ◎ |
| .vscode/extensions.json(推奨拡張) | − | − | ◎ |
| .vscode/settings.json(Git管理) | − | − | ◎ |
| **■ 同期・再現の単位** | | | |
| 再現方法 | Settings Syncでアカウント同期 | なし(何も持たない) | git clone → Reopen in Container |
| 対象範囲 | 個人(自分の全PC) | − | チーム全員 |

### 表の読み方(要点)

1. **Windows列に◎が付くのは「見た目・操作・接続」だけ**
   - 言語系拡張(Volar、ESLint等)はWindows側に入れない。
     リモート接続中は機能しないか、二重に動いて混乱の元になる
2. **WSL2列はほぼ空** — 開発の実体はコンテナ内のため、WSLは薄く保つ
3. **開発に関わるものはすべてコンテナ列でGit管理** — この縦の分離が
   崩れなければ環境の再現性が保たれる

## 管理方法: 3段構え

### 1. devcontainer.json — プロジェクト必須拡張(チーム強制)

コンテナ接続時に自動インストールされる拡張を定義。**Gitでプロジェクトと
一緒に管理**する。この構成の中核。

```json
{
  "name": "vue-capacitor-dev",
  "image": "myapp:jdk21-node22",
  "forwardPorts": [5173],
  "customizations": {
    "vscode": {
      "extensions": [
        "Vue.volar",
        "dbaeumer.vscode-eslint",
        "esbenp.prettier-vscode",
        "EditorConfig.EditorConfig"
      ],
      "settings": {
        "editor.formatOnSave": true,
        "editor.defaultFormatter": "esbenp.prettier-vscode",
        "files.eol": "\n"
      }
    }
  }
}
```

メリット:

- チーム全員・全マシンで同じ開発環境が再現される
  (新メンバーは「Reopen in Container」するだけ)
- 拡張の入れ忘れ・バージョン差異による「自分の環境だけLintが効かない」が消える
- `settings` も書けるため、フォーマッタ設定等もコード化できる

### 2. Settings Sync — 個人の好み(全環境共通)

VS Code標準の設定同期(GitHub/Microsoftアカウント)で、**Windows側の
テーマ・キーバインド・UI設定**を同期する。

- 対象: テーマ、キーマップ、UI系拡張、個人スニペット
- 複数PC(会社・自宅)で同じ使い勝手になる
- **プロジェクト固有の設定はここに入れない**(個人とプロジェクトの分離が原則)

### 3. .vscode/extensions.json — チーム「推奨」(強制しない)

必須ではないが推奨したい拡張は、リポジトリの `.vscode/extensions.json` に
記載すると、開いたときに「推奨拡張があります」と通知される。

```json
{
  "recommendations": [
    "eamodio.gitlens",
    "streetsidesoftware.code-spell-checker"
  ]
}
```

> 使い分け: **必須は devcontainer.json、あると便利は extensions.json**

### 設定(settings)の優先度

下に行くほど優先度が高い。

| レイヤ | 保存場所 | 用途 |
|---|---|---|
| User(Windows) | Windowsのユーザー設定 | 個人の好み(Settings Syncで同期) |
| Remote(コンテナ) | devcontainer.jsonの `settings` | コンテナ内共通の既定値 |
| Workspace | `.vscode/settings.json`(Git管理) | プロジェクト固有設定(最優先) |

## 拡張機能の導入方式: イメージには焼き込まない

### 標準方式: 動的インストール(推奨)

devcontainer.json の拡張は、**コンテナ起動後にVS Codeが実行中のコンテナへ
動的にインストール**する。イメージに焼き込まれるわけではない。

「Reopen in Container」時の動き:

```
1. コンテナ起動(イメージは素のまま: JDK21+Node22+SDK)
2. VS Code が VS Code Server をコンテナ内に自動インストール
     → 置き場所: ~/.vscode-server/(コンテナ内)
3. devcontainer.json の extensions に列挙された拡張を
   Marketplace からダウンロードしてコンテナ内にインストール
4. 以後、言語解析などはコンテナ内の Server で動作
```

ポイント:

- **イメージとVS Code環境が分離** — イメージはビルドツールチェーンだけを持ち、
  エディタ環境は接続時に構成される
- 拡張は `~/.vscode-server/extensions/` に入るため、
  **コンテナを破棄して作り直すと再ダウンロード**(数分程度)。
  同じコンテナを使い続ける限り再インストールは発生しない
- 拡張の追加・変更は devcontainer.json の1行編集で済み、
  **イメージの再ビルド・再配布が不要**

### 責任分離の原則

> **イメージは「ビルドできる環境」、devcontainer.json は「開発できる環境」**

| 項目 | 内容 |
|---|---|
| イメージの中身 | JDK 21 + Node 22 + Android SDK のみ(現行計画のまま) |
| 拡張の管理 | devcontainer.json(Git管理) |

## プロキシ環境での注意点(重要)

接続時に発生するダウンロード:

| ダウンロード対象 | 取得元 |
|---|---|
| VS Code Server本体 | update.code.visualstudio.com |
| 拡張機能(.vsix) | Visual Studio Marketplace |

コンテナ内からこれらに到達できないと接続に失敗する。対処の順序:

1. **コンテナへのプロキシ環境変数注入**
   (`~/.docker/config.json` の `proxies.default`)が効いていれば通常そのまま通る
2. 拡張のダウンロードだけ失敗する場合、**Windows側でダウンロードして
   コンテナに転送する**方式に切り替え(Windows側はPACプロキシで外に出られる):

   ```json
   // Windows側のUser設定(settings.json)
   {
     "remote.downloadExtensionsLocally": true
   }
   ```

## プリベイク(拡張入りイメージ)を検討するケース

以下に当てはまる場合のみ検討する:

- プロキシ/ネットワーク制約が厳しく、コンテナからMarketplaceに到達できない
  (かつ `downloadExtensionsLocally` でも運用が苦しい)
- コンテナを頻繁に作り直す運用(CI上の検証等)で毎回の拡張DLが遅すぎる
- オフライン環境への配布が必要

### 方法a: Dev Container CLI でプリビルド

```bash
npm install -g @devcontainers/cli
devcontainer build --workspace-folder . --image-name myapp-dev:prebuilt
```

devcontainer.json の内容(featuresや一部設定)を反映したイメージを事前生成
できる。ただし拡張機能自体の完全な焼き込みは対象外の部分がある。

### 方法b: Dockerfile で拡張を直接インストール

VS Code Server のバージョンに依存するため壊れやすく、メンテナンスコストが
高い。**最終手段**。

## まとめ: 運用イメージ

```
個人の好み      → Settings Sync(アカウント同期)
                   └ Windows側: テーマ、キーマップ、Dev Containers拡張

プロジェクト必須 → Git管理
                   ├ .devcontainer/devcontainer.json(拡張+コンテナ設定)
                   ├ .vscode/settings.json(フォーマット等の強制設定)
                   └ .vscode/extensions.json(推奨拡張)

WSL2(Ubuntu)   → 何も入れない(コンテナの土台に徹する)

イメージ        → ビルドツールチェーンのみ(拡張は焼き込まない)
```

この形にしておくと、PCを変えても、メンバーが増えても、
**`git clone` → 「Reopen in Container」で同じ環境が立ち上がる**状態になる。
プロキシ起因の問題が出た場合のみ、`downloadExtensionsLocally` →
プリベイクの順に回避策を適用する。
