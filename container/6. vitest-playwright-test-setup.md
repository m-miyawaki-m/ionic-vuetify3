# テスト環境の事前準備(Vitest + Playwright)

## このドキュメントの目的

Vue + Vuetify 4 + Capacitor プロジェクトに Vitest(単体/コンポーネントテスト)
と Playwright(E2Eテスト)を導入する際に、**コードを書き始める前に
整備しておくべきこと**をまとめる。

テストは後付けすると「テストできない構造」が先にできてしまうため、
プロジェクト初期に環境と規約を固めることが重要。

---

## 0. テスト戦略の全体像

| レイヤ | ツール | 対象 | 実行場所 |
|---|---|---|---|
| 単体テスト | **Vitest** | composables、services、stores、ユーティリティ | コンテナ |
| コンポーネントテスト | **Vitest** + Vue Test Utils | 共通部品(AppXxx)、画面の表示分岐 | コンテナ |
| E2Eテスト | **Playwright** | 画面をまたぐユーザーフロー | コンテナ(要追加対応)or Windows |
| ネイティブ機能 | (自動化対象外) | カメラ・通知等の実機挙動 | Windowsエミュレータで手動確認 |

方針:

- **テストピラミッド**: 数は 単体 > コンポーネント > E2E。
  E2Eは「主要フローの保証」に絞り、細かい分岐は単体で担保する
- **ネイティブ機能はE2E自動化しない**: PlaywrightはブラウザのE2Eであり、
  Capacitorのネイティブブリッジは対象外。services/native層をモックして
  Web側のロジックをテストし、実機挙動はエミュレータで手動確認する
  (実装規約⑤で services/native ラップを必須にしたのはこのため)

---

## 1. Vitest の事前準備

### 1-1. インストール

```bash
npm install -D vitest @vue/test-utils happy-dom @vitest/coverage-v8
npm install -D vitest-canvas-mock   # Vuetifyの一部コンポーネント対策
```

- `happy-dom`(または `jsdom`): Node上でDOMを再現する環境。
  コンポーネントテストに必須
- `@vitest/coverage-v8`: カバレッジ計測

### 1-2. 設定ファイル

Viteの設定を継承する形で `vitest.config.ts` を作る:

```typescript
// vitest.config.ts
import { defineConfig, mergeConfig } from 'vitest/config'
import viteConfig from './vite.config'

export default mergeConfig(viteConfig, defineConfig({
  test: {
    environment: 'happy-dom',
    globals: true,                    // describe/it/expect をimport不要に
    setupFiles: ['./vitest.setup.ts'],
    server: {
      deps: {
        inline: ['vuetify'],          // ★Vuetify対応(重要・後述)
      },
    },
    coverage: {
      provider: 'v8',
      include: ['src/**'],
      exclude: ['src/main.ts', 'src/**/*.d.ts'],
    },
  },
}))
```

### 1-3. Vuetifyコンポーネントをテストするための特殊設定(重要)

Vuetifyコンポーネントを含むテストは、素のVue Test Utilsだけでは動かない。
以下の準備が**事前に**必要:

**① セットアップファイルでブラウザAPIをモックする**

VuetifyはResizeObserver等のブラウザAPIを使うが、happy-dom/jsdomには
存在しない(または不完全)ため:

```typescript
// vitest.setup.ts
import { vi } from 'vitest'

// ResizeObserver モック(Vuetifyレイアウト系で必須)
global.ResizeObserver = vi.fn().mockImplementation(() => ({
  observe: vi.fn(),
  unobserve: vi.fn(),
  disconnect: vi.fn(),
}))

// visualViewport モック(ダイアログ系で必要になることがある)
Object.defineProperty(window, 'visualViewport', {
  value: { addEventListener: vi.fn(), removeEventListener: vi.fn() },
})

// matchMedia モック(テーマ'system'判定・ブレークポイントで必要)
Object.defineProperty(window, 'matchMedia', {
  value: vi.fn().mockImplementation((query: string) => ({
    matches: false, media: query,
    addEventListener: vi.fn(), removeEventListener: vi.fn(),
  })),
})
```

**② テスト用のマウントヘルパーを用意する**

毎回Vuetifyインスタンスを作るのは冗長なので、共通ヘルパーを最初に作る:

```typescript
// tests/helpers/mount.ts
import { mount, type ComponentMountingOptions } from '@vue/test-utils'
import { createVuetify } from 'vuetify'
import * as components from 'vuetify/components'
import * as directives from 'vuetify/directives'

const vuetify = createVuetify({ components, directives })

export function mountWithVuetify<T>(
  component: T,
  options: ComponentMountingOptions<T> = {},
) {
  return mount(component, {
    ...options,
    global: {
      plugins: [vuetify],
      ...options.global,
    },
  })
}
```

**③ `server.deps.inline: ['vuetify']`**(1-2の設定)

VuetifyはESM/CSSの取り回しが特殊なため、Vitestに変換させる指定。
これがないと `Unknown file extension ".css"` 等のエラーになる。

### 1-4. Capacitorプラグインのモック方針

services/native/ 経由でラップしている前提(実装規約⑤)なので、
**テストではservices層をモックする**のが基本:

```typescript
// 例: カメラを使うcomposableのテスト
import { vi } from 'vitest'

vi.mock('@/services/native/camera', () => ({
  takePhoto: vi.fn().mockResolvedValue('blob:dummy-photo-url'),
}))
```

services/native/ 自体をテストしたい場合のみ `@capacitor/*` をモックする:

```typescript
vi.mock('@capacitor/core', () => ({
  Capacitor: { isNativePlatform: () => false },
}))
```

### 1-5. 規約(事前に決めておくこと)

- テストファイルの置き場所: **対象の隣に `*.spec.ts`**
  (例: `useConfirm.ts` と同階層に `useConfirm.spec.ts`)
  ※ E2Eと区別するため単体は `.spec.ts`、E2Eは `tests/e2e/` 配下に統一
- package.json:

  ```json
  {
    "scripts": {
      "test:unit": "vitest run",
      "test:unit:watch": "vitest",
      "test:coverage": "vitest run --coverage"
    }
  }
  ```

- VS Code拡張「Vitest」(`vitest.explorer`)をdevcontainer.jsonに追加すると
  エディタ内でテスト実行・デバッグができる

## 2. Playwright の事前準備

### 2-1. 重要: 実行場所の決定(コンテナの制約)

**Playwrightのブラウザ自動インストール(`--with-deps`)は
Ubuntu/Debian系が前提**で、Oracle Linux 9(RHEL系)コンテナは
公式サポート外。事前にどこで動かすか決めておく必要がある。

| 案 | 内容 | 評価 |
|---|---|---|
| a. 公式Playwrightコンテナを併用 | E2E実行時だけ `mcr.microsoft.com/playwright` イメージを使う | **推奨**。依存問題が消える |
| b. Windows側で実行 | Windowsにnode+Playwrightを入れ、コンテナのdevサーバーに接続 | 手軽だが環境統一方針から外れる |
| c. Oracle Linux 9に依存ライブラリを手動導入 | dnfで必要ライブラリを個別インストール | 茨の道。バージョンアップごとに苦労 |

**推奨は案a**: 開発用イメージはそのまま、E2E実行は公式イメージで行う。
docker-compose等で並べる:

```yaml
# compose.e2e.yaml(イメージ例)
services:
  e2e:
    image: mcr.microsoft.com/playwright:v1.x.x-noble   # ←Playwrightと同じversionに揃える(重要)
    working_dir: /work
    volumes: [ ".:/work" ]
    network_mode: host        # devサーバーへlocalhostで届くように
    command: npx playwright test
```

> **バージョン一致の原則**: package.json の `@playwright/test` のバージョンと
> 公式イメージのタグは必ず一致させる。ズレるとブラウザ不一致エラーになる。

### 2-2. インストールと初期化

```bash
npm init playwright@latest
# → tests/e2e/ 配下に置く、TypeScript、GitHub Actionsは必要に応じて
```

プロキシ環境ではブラウザのダウンロードが失敗しやすい。
`HTTPS_PROXY` が効いていることを確認の上:

```bash
npx playwright install chromium
```

(社内事情で直接DLできない場合は `PLAYWRIGHT_DOWNLOAD_HOST` で
社内ミラーを指定する方法もある)

### 2-3. 設定ファイル(モバイル前提の構成)

ハイブリッドアプリなので、**モバイルビューポートでのテストを既定**にする:

```typescript
// playwright.config.ts
import { defineConfig, devices } from '@playwright/test'

export default defineConfig({
  testDir: './tests/e2e',
  use: {
    baseURL: 'http://localhost:5173',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
  },
  projects: [
    {
      name: 'mobile-chrome',
      use: { ...devices['Pixel 7'] },     // モバイル相当を既定に
    },
    {
      name: 'desktop-chrome',
      use: { ...devices['Desktop Chrome'] },  // 必要なら併用
    },
  ],
  webServer: {
    command: 'npm run dev -- --host 0.0.0.0',
    url: 'http://localhost:5173',
    reuseExistingServer: true,   // 起動済みdevサーバーがあれば再利用
  },
})
```

ポイント:

- **`webServer` 設定**: テスト実行時にdevサーバーを自動起動してくれる。
  CIでもローカルでも同じコマンドで動く
- **`devices['Pixel 7']`**: ビューポート・UA・タッチ操作をモバイル相当に
  エミュレート。アプリの実態(WebView表示)に近い条件でテストできる

### 2-4. テストを書きやすくするための事前整備

**① data-testid の規約を決める**

VuetifyはDOM構造が複雑で、クラス名でのセレクタは壊れやすい。
**操作対象には `data-testid` を付ける**規約を実装側に最初から入れる:

```vue
<v-btn data-testid="login-submit" @click="login">ログイン</v-btn>
```

```typescript
// テスト側
await page.getByTestId('login-submit').click()
```

(セマンティックに取れるもの — role, label — はそちらを優先し、
testidは最後の手段とするのが公式推奨。ただし規約として用意はしておく)

**② ネイティブ機能のWeb フォールバックを用意しておく**

PlaywrightはブラウザでのE2Eのため、`Capacitor.isNativePlatform()` は
常に false。実装規約⑤の「Webフォールバック必須」が守られていれば、
E2Eはフォールバック経路でフローを通せる。守られていないと
E2Eが書けない画面が生まれる。

**③ package.json スクリプト**

```json
{
  "scripts": {
    "test:e2e": "playwright test",
    "test:e2e:ui": "playwright test --ui",
    "test:e2e:report": "playwright show-report"
  }
}
```

`--ui` モード(GUIでステップ実行)はWindows側ブラウザで開く形になるため、
コンテナ実行時はポートフォワードまたはWindows側実行を使う。
