# Vue + Vuetify 4 + Capacitor モバイルアプリ開発環境構築

## 結論: 2つの環境による役割分担が必須

1台のWindows PC上に、役割の異なる**2つの環境を併存**させる構成とする。

| 環境 | 役割 |
|---|---|
| **Linux環境**(WSL2上のコンテナ: Oracle Linux 9 + JDK 21 + Node 22) | ソースコード管理、Vite開発サーバー、Gradle + Android SDKによるAPKビルド |
| **Windows環境** | Android Studio(エミュレータ管理・起動用)、VS Code、ブラウザ |

### なぜコンテナに全部入れられないのか

Androidエミュレータをコンテナ内で動かすのは技術的には不可能ではないが、**推奨しない**。

- エミュレータには `/dev/kvm`(ハードウェア仮想化)が必須
  - WSL2では `nestedVirtualization=true` + `--device /dev/kvm` で渡せるが、
    「Windows(Hyper-V)→ WSL2 → KVM → エミュレータ」の**二重仮想化**になり性能が明確に劣化
- GUI表示にはWSLgソケットのコンテナへのマウントが必要で、IME・HiDPI等の問題が頻発
- Googleのサポート対象外構成であり、アップデートで突然壊れるリスク
- イメージが数GB級に肥大化、AVD永続化のボリューム設計も必要
- 得られるメリットに対してコスト(構築・維持・性能)が見合わない

### Android Studio の要素分解

「Android Studio」は3要素の集合であり、フルIDEが必要なのはエミュレータ管理のみ。

| 要素 | 必要性 | 配置 |
|---|---|---|
| IDE(エディタ機能) | 不要(VS Codeで編集) | — |
| Android SDK / Gradleビルド | 必要 | **コンテナ** |
| エミュレータ(AVD) | 必要 | **Windows** |

## 全体アーキテクチャ

```
[Windows]
 ├─ VS Code(エディタ本体)
 │    └─ Dev Containers拡張 ──→ コンテナ内のソースを直接編集
 ├─ Android Studio + AVD(Android 13 / API 33エミュレータ)
 └─ ブラウザ(Chrome DevTools デバイスモード)← 日常の確認はここ
        ↑ localhost:5173
[WSL2 / コンテナ(Oracle Linux 9 + JDK21 + Node22)]
 ├─ Vue + Vuetify 4 ソースコード
 ├─ Vite開発サーバー(ホットリロード)
 └─ Gradle + Android SDK cmdline-tools(APKビルド)
        ↓ ビルドしたAPK
[Windowsのエミュレータへ adb install / Capacitorライブリロード]
```

### 作業の役割分担

| 作業 | 実行場所 | 補足 |
|---|---|---|
| ソース編集 | VS Code(Windows)+ Dev Containers | コンテナ内ファイルを直接編集 |
| 開発サーバー(npm run dev) | コンテナ | Viteのホットリロード |
| 日常のUI確認 | Chromeデバイスモード | 最速のフィードバックループ |
| APKビルド | コンテナ | ツールチェーンを統一、CI/CD移行も容易 |
| エミュレータ実行 | Windows(Android Studio AVD) | コンテナでは実行不可 |
| ネイティブ機能確認 | Windowsエミュレータ | カメラ・通知等の確認時のみ |

### 開発ワークフロー方針

- **コード修正の9割はブラウザのデバイスモードで確認**(最速ループ)
- ブラウザで確認できないもの(ネイティブプラグイン、戻るボタン挙動、
  セーフエリア、実機性能)だけエミュレータで確認

## 重要: Capacitor のバージョンと JDK の対応

| Capacitor | 必要なJDK |
|---|---|
| Capacitor 6 | **Java 17** |
| Capacitor 7 | **Java 21** |

- 用意したコンテナは **JDK 21** のため、整合性を取るなら **Capacitor 7 を推奨**
- Vuetify 4 は2026年2月に正式リリース済み(最新 v4.1.0)で問題なし
- Capacitor 6 固定の事情がある場合は、コンテナをJDK 17のままにするか、
  17/21併存にして Gradle toolchain で切り替える

## 構築手順

### Step 1: コンテナへ Android SDK cmdline-tools を追加

```dockerfile
ENV ANDROID_HOME=/opt/android-sdk
RUN mkdir -p $ANDROID_HOME/cmdline-tools && \
    curl -o /tmp/cmdtools.zip https://dl.google.com/android/repository/commandlinetools-linux-11076708_latest.zip && \
    unzip /tmp/cmdtools.zip -d $ANDROID_HOME/cmdline-tools && \
    mv $ANDROID_HOME/cmdline-tools/cmdline-tools $ANDROID_HOME/cmdline-tools/latest && \
    rm /tmp/cmdtools.zip
ENV PATH=$PATH:$ANDROID_HOME/cmdline-tools/latest/bin:$ANDROID_HOME/platform-tools
RUN yes | sdkmanager --licenses && \
    sdkmanager "platform-tools" "platforms;android-34" "build-tools;34.0.0"
```

※ プロキシ環境では `sdkmanager --proxy=http --proxy_host=... --proxy_port=...` が必要な場合あり

### Step 2: プロジェクト作成(コンテナ内)

```bash
npm create vue@latest my-app
cd my-app
npm install vuetify
npm install @capacitor/core @capacitor/android
npm install -D @capacitor/cli
npx cap init my-app com.example.myapp --web-dir=dist
npx cap add android
```

### Step 3: VS Code の接続(Windows側)

1. VS Code に「Dev Containers」拡張をインストール
2. `.devcontainer/devcontainer.json` を用意:

```json
{
  "name": "vue-capacitor-dev",
  "image": "myapp:jdk21-node22",
  "forwardPorts": [5173],
  "customizations": {
    "vscode": {
      "extensions": ["Vue.volar", "dbaeumer.vscode-eslint"]
    }
  }
}
```

3. 「Reopen in Container」でコンテナ内ソースを直接編集
4. ソースはWSLファイルシステム内(`~/projects/...`)に置く(`/mnt/c` は遅い)

### Step 4: 日常の開発フロー(ブラウザ確認)

```bash
npm run dev -- --host 0.0.0.0
```

- Windowsのブラウザで `http://localhost:5173`(WSL2のlocalhost転送が効く)
- Chrome DevTools のデバイスモード(F12 → デバイスツールバー)で
  スマホ画面サイズ・タッチ操作をシミュレート

### Step 5: エミュレータでの確認(Windows側)

**準備**: Android Studio で AVD(Pixel系 / Android 13 = API 33)を作成・起動

#### 方法a: APKビルド → インストール(リリースに近い確認)

```bash
# コンテナ内
npm run build
npx cap sync android
cd android && ./gradlew assembleDebug
# → android/app/build/outputs/apk/debug/app-debug.apk
```

APKをWindows側から見える場所に置き、Windowsのコマンドプロンプトで:

```cmd
adb install app-debug.apk
```

#### 方法b: Capacitor ライブリロード(開発中の確認に最適)

`capacitor.config.ts`:

```typescript
const config: CapacitorConfig = {
  // ...
  server: {
    url: 'http://10.0.2.2:5173',  // エミュレータから見たWindowsホスト
    cleartext: true
  }
};
```

- エミュレータから `10.0.2.2` = Windowsのlocalhost → WSL2のlocalhost転送経由で
  コンテナのViteに到達
- **この設定はコミットしない**(本番ビルドに混入するとアプリが動かない)

## リモート開発への発展性

コンテナ側(Linux環境)にビルドツールチェーンを集約しているため:

- 将来、開発サーバーを別のLinuxマシン(社内サーバー、クラウド)に移す場合も
  **同じコンテナイメージをそのまま移植**できる
- その場合もVS Code(Remote - SSH + Dev Containers)とWindowsのエミュレータの
  構図は変わらない(ライブリロードのURLがリモートホスト宛になる点のみ変更)
- CI/CDパイプラインでも同一イメージでAPKビルドが可能

---

## 注意点

- **Capacitorのcleartext通信**: ライブリロードはhttpのため、`cleartext: true` と
  Android側の `usesCleartextTraffic` 許可が必要(開発時のみ)
- **Viteの起動オプション**: `--host 0.0.0.0` での起動が必須(WSLの
  ネットワークモードによらず共通)
- **エミュレータのアクセラレーション**: WSL2利用時点でHyper-V基盤は有効のため、
  Windows側エミュレータは通常問題なく高速動作する
- **adbのバージョン不一致**: コンテナ内とWindows側のplatform-toolsのバージョン差が
  大きいと接続が不安定になることがある。基本は「APK受け渡し+Windows側で
  adb install」がトラブルが少ない
- **Windows側を軽くしたい場合**: Android Studioなしで cmdline-tools のみで
  エミュレータ運用も可能だが、AVD管理はGUIが楽なため、素直にAndroid Studioを
  入れるのが結局手間が少ない

  ```cmd
  sdkmanager "emulator" "system-images;android-33;google_apis;x86_64" "platform-tools"
  avdmanager create avd -n pixel_api33 -k "system-images;android-33;google_apis;x86_64"
  emulator -avd pixel_api33
  ```
